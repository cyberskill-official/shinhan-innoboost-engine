# Feature Request

> Turn Your Will Into Real.

## Summary

Implement the GitHub Actions CI/CD pipeline that lints, type-checks, unit-tests, integration-tests, runs the eval harness, scans for security issues (Trivy + Snyk + Semgrep + CodeQL + secret-scanning), builds and signs container images with Sigstore cosign, attaches SBOMs generated by Syft, attaches SLSA Level 3 provenance attestation on releases, pushes images to GHCR (GitHub Container Registry), and deploys to staging on merge to `main`. Five workflows total: `pr-check`, `main-deploy`, `release`, `security-scan`, `eval-harness`. Every PR must pass the full pipeline. Every release tag must produce a signed, SBOM-attached, provenance-attested container set. The pipeline is the single most important piece of operational-maturity evidence we will show Shinhan reviewers, and it is the structural enforcement of every "we follow best practices" claim in the pitch.

## Problem

A demo that ships from "works on my laptop" is not production-ready and will fail any banking-sector security review. We need automated gates between developer intent and any environment a Shinhan reviewer might touch — including the staging environment used for rehearsal demos.

Specific gaps if we skip or shortcut:

- **Without security scans on every PR, vulnerable dependencies leak into staging and into the demo container.** Shinhan reviewers may inspect the dependency manifest; an unpatched Critical CVE makes the rest of the demo irrelevant.
- **Without SBOM and provenance attestation, Phase 8 compliance work has no evidence chain.** ISO 27001 control A.8.30 (outsourced development) and SOC 2 CC6.6 (vulnerability management) both require evidence; SBOM + attestation feeds that evidence directly.
- **Without push-to-GHCR-on-tag, on-prem deployment requires manual image curation.** Manual curation introduces drift and the "but on my machine" failure mode at the worst time.
- **Without auto-deploy-to-staging, Phase 12 rehearsal has no stable target.** Every rehearsal becomes a re-deploy operation, which is exactly the wrong way to spend rehearsal time.
- **Without secret-scanning, a leaked Anthropic API key or an embedded Claude system-prompt could go live.** Both are credibility-destroying events; preventable with cheap tooling.

The Innoboost Q&A flags that Shinhan reviewers will press on operational maturity (Section V.4 — "commercial readiness and market traction"). A real CI/CD pipeline with signed images, SBOMs, and provenance is the cheapest signal of operational maturity we can give them. It is also the easiest claim to *demonstrate* live: pull up a recent pipeline run during the interview, walk through the security gates, point at the SBOM. Specific is better than generic; visible is better than claimed.

The `feedback_p1_scope_preference` memory note biases us toward the richer scope. For CI/CD, "richer" means including SLSA Level 3 provenance from the start — slightly more setup cost, dramatically stronger compliance posture. The cost delta is small (a few hundred lines of YAML, a one-time Sigstore key setup); the credibility delta is large.

There is a downstream-multiplier dimension: every Phase 2+ task ships through this pipeline. A slow pipeline (e.g., 20-minute PR-check) costs the team many engineer-hours per week in waiting. A fast pipeline (5-minute PR-check) is force-multiplying. Optimising the pipeline up-front is high-leverage.

## Proposed Solution

A GitHub Actions pipeline at `.github/workflows/` with five workflows. Container images signed with Sigstore cosign (keyless OIDC); SBOMs generated by Syft; vulnerability scans by Trivy; dependency scans by Snyk + Dependabot; static-analysis scans by Semgrep + CodeQL; secret-scanning by GitHub-native + TruffleHog. Images pushed to `ghcr.io/cyberskill-official/shinhan-innoboost/*`. Staging deploy via GitHub Environments with required reviewers. Release pipeline produces SLSA Level 3 provenance attestations attached to every release artefact. Pipeline runtime target: PR-check median < 8 minutes; release pipeline < 15 minutes.

### Subtasks

- [ ] **Workflow 1: `pr-check.yml`.** Triggers on every PR. Jobs (in parallel where possible):
  - `lint`: pnpm install (with cache); `pnpm turbo run lint`. Fail on any error.
  - `typecheck`: `pnpm turbo run typecheck`. Fail on any error.
  - `test`: `pnpm turbo run test --concurrency=4`. Fail on any test failure; report coverage to Codecov (90% target on engine workspace).
  - `build`: `pnpm turbo run build`. Fail on any build error.
  - `containerise`: build Docker images for each workspace via `pnpm turbo run containerise`. Fail on any container build error.
  - `trivy-scan`: scan each built image for HIGH and CRITICAL CVEs; comment scan results to PR. Fail on Critical; warn on High.
  - `semgrep-scan`: run Semgrep with custom CyberSkill ruleset (P02-T06 prompt-injection rules from when ratified, plus standard OWASP Top 10 rules). Comment results.
  - `secret-scan`: run TruffleHog on the diff. Fail on any matched secret pattern.
  - `commit-validation`: re-run commitlint against the PR commits (in case of a force-push that bypassed local Husky).
- [ ] **Workflow 2: `main-deploy.yml`.** Triggers on push to `main` (which only happens via merged PR). Jobs:
  - Re-run the full `pr-check` for safety (cached cleanly).
  - `build-and-push`: build all workspace images; sign with cosign keyless OIDC; attach SBOM via Syft; push to GHCR with tags `:main-{sha}` and `:latest`.
  - `deploy-staging`: deploy to GitHub Environment `staging` via `helm upgrade --install` against the staging Kubernetes cluster (cluster credentials from GitHub OIDC federation, not long-lived secrets).
  - `smoke-test-staging`: run a 3-minute smoke-test suite against the deployed staging environment; verify engine health endpoint returns 200; verify gold-set sample (5 questions) produces expected answers.
  - `notify-slack`: post deployment status to `#shinhan-innoboost-2026` channel with image tag and smoke-test result summary.
- [ ] **Workflow 3: `release.yml`.** Triggers on push of tag `v*.*.*`. Jobs:
  - Run the full `main-deploy` flow.
  - `provenance`: generate SLSA Level 3 provenance attestation using `slsa-github-generator`.
  - `release-images`: push images with `:v{x.y.z}` tag in addition to `:main-{sha}` and `:latest`.
  - `release-notes`: auto-generate release notes from conventional commits since the previous tag.
  - `github-release`: create a GitHub Release with SBOM, provenance attestation, and image-list as assets; release notes from above; mark as latest.
- [ ] **Workflow 4: `security-scan.yml`.** Triggers nightly on `main`. Jobs:
  - `trivy-full`: scan every published image (most-recent on each tag) and report findings to GitHub Advanced Security.
  - `snyk-deep`: run Snyk's deep-dependency-scan (different pattern than per-PR Trivy); upload SARIF to GitHub Code Scanning.
  - `semgrep-rules-update`: pull latest Semgrep rule updates; re-scan codebase; comment any new findings as auto-created issues.
  - `codeql-full`: GitHub-native CodeQL run; upload SARIF.
  - `gpu-cost-check`: query Lambda Labs API for spend; alert if over $300/week (ties into P00-T05 cost discipline).
  - Auto-create issues for every new HIGH or CRITICAL finding; assign to engineering tech lead.
- [ ] **Workflow 5: `eval-harness.yml`.** Triggers on PRs touching `engine/`, `eval/`, or any prompt files; also nightly on `main`. Jobs:
  - `eval-incremental` on PR: detect changed metrics; run only their gold-set; compare to baseline; comment delta to PR; block merge on regression > 2%.
  - `eval-full` nightly on `main`: run the entire gold-set; produce dashboard update; alert on regression > 1%.
  - Detailed implementation lives in P04-T04; this task only sets up the workflow scaffold and trigger conditions.
- [ ] **GitHub Environments configuration.** Configure three environments:
  - `staging`: auto-deploy on `main`; protection rules: secret scoping (only staging secrets); URL recorded for the rehearsal team.
  - `production-rehearsal`: manual approval required (founder + engine tech lead); used for day-before-demo locked builds.
  - `production-shinhan-poc`: placeholder; fully locked until kickoff; cannot be deployed to.
- [ ] **Secrets management.** Configure GitHub secrets at the appropriate scope (org-level vs. repo-level vs. environment-level):
  - `GHCR_TOKEN`: org-level; for image push.
  - `SLACK_WEBHOOK`: repo-level; for notifications.
  - `SNYK_TOKEN`: repo-level; for scanning.
  - `STAGING_KUBECONFIG`: environment-level (staging only); rotated quarterly; preferably replaced by OIDC federation to GKE/EKS so we don't store long-lived kubeconfig.
  - `SLSA_OIDC_TOKEN`: environment-level; for keyless cosign signing.
- [ ] **OIDC federation to cloud providers.** Configure GitHub OIDC ↔ GCP Workload Identity (or AWS IAM Roles for Service Accounts) so that the deploy workflow gets short-lived cloud credentials per run rather than holding long-lived kubeconfig in secrets. Strong security posture; aligns with banking-sector expectations.
- [ ] **Renovate compatibility.** Configure Renovate to *not* bypass branch protection; security-priority patches still go through `pr-check`. The exception: if a vulnerability is rated Critical with active exploitation, document an emergency-merge process (founder + engineering tech lead 2-eye approval, audit-logged) — but never silent bypass.
- [ ] **Required status checks.** Set on `main`: `pr-check / lint`, `pr-check / typecheck`, `pr-check / test`, `pr-check / build`, `pr-check / containerise`, `pr-check / trivy-scan`, `pr-check / semgrep-scan`, `pr-check / secret-scan`, `pr-check / commit-validation`. The latest nightly `security-scan` must be green-or-acknowledged.
- [ ] **CI runtime budget.** Tune cache hits (pnpm store, Turborepo Remote Cache, Docker layer cache) to keep median PR-check time under 8 minutes. Profile slow steps, optimise.
- [ ] **Author runbook.** `docs/runbooks/cicd.md`: how to debug failing CI; how to bypass for emergency hotfixes (with audit-log requirements); how to roll back staging; how to retrigger a failed deploy; how to read SBOM and provenance.
- [ ] **Compliance one-pager.** A one-page CI/CD spec for the Shinhan compliance dossier (P11-T04). Visual: pipeline diagram. Text: gates, evidence produced, retention. Audience: Shinhan compliance reviewer who has 5 minutes.

### Acceptance criteria

- All five workflows operational and passing on the Phase-1 skeleton (P01-T01).
- Container images signed with cosign keyless OIDC; SBOMs attached; verified by `cosign verify` against a sample image.
- Staging environment auto-deploys from `main`; smoke-test passes; engine health endpoint reachable.
- Required status checks enforced on `main` branch protection; verified by attempting a PR-check failure (should block merge).
- SLSA Level 3 provenance attestation produced on a sample release tag and verifiable by `slsa-verifier`.
- Runbook published; compliance one-pager in the dossier folder.
- Median PR-check runtime < 8 minutes (measured over the first 10 PRs after pipeline activation).
- Secret-scan does not flag any false-positive on the skeleton; no real secret detected.

## Alternatives Considered

- **GitLab CI instead of GitHub Actions.** Rejected: `cyberskill_company_facts` locks GitHub as the SCM; switching SCM for one project is irrational.
- **Buildkite or CircleCI.** Rejected: adds cost and tool-sprawl; GitHub Actions is sufficient for this scope and integrates natively with the Security tab.
- **Skip SBOM/provenance for the demo phase, add later.** Rejected: adding compliance evidence retroactively means a cleanup sprint. Do it once, do it right, from the start.
- **Self-hosted runners.** Rejected for the demo phase — GitHub-hosted runners are sufficient and avoid an infra-management task we don't need now. Revisit for production-track if Shinhan requires runner-level egress controls.
- **Skip Sigstore in favour of long-lived signing keys.** Rejected: long-lived keys are a worse security posture; keyless OIDC is now standard. Setup cost is similar; ongoing maintenance much lower.
- **Skip SLSA Level 3; ship at Level 2.** Rejected: the cost delta is small; the credibility delta in front of a banking compliance team is large. Level 3 also fits the `feedback_p1_scope_preference` bias.
- **Use Docker Hub instead of GHCR.** Rejected: GHCR is closer to the source repo, integrates natively with GitHub OIDC, and avoids Docker Hub's pull-rate complications for on-prem deployment scenarios.
- **Use Argo CD or Flux for GitOps deploy instead of `helm upgrade`.** Rejected for the demo phase: GitOps adds complexity disproportionate to the simple staging environment we have. Reconsider for production track if Shinhan operates GitOps-shaped infrastructure.

- Q3: For SLSA provenance, do we use `slsa-github-generator` (community-maintained) or build our own attestor? Recommendation: community-maintained. Less burden, well-vetted.
- Q4: Should every workspace's image be published, or only the deployable ones (engine, hitl, ui)? Recommendation: only deployable; the eval/, data/, infra/, compliance/ workspaces produce non-runtime artefacts (eval reports, generators, IaC, docs).

## Implementation Notes

- Use `actions/cache` for pnpm store + Turborepo cache + Docker layer cache. Cache invalidation: pnpm based on `pnpm-lock.yaml` hash; Turborepo based on its own keys; Docker based on Dockerfile + COPY-source hashes.
- The cosign keyless OIDC flow uses GitHub's OIDC token; no key management on our side. Verification by `cosign verify --certificate-identity-regexp` matching our repo.
- For SLSA Level 3, the provenance attestation must be unforgeable — the `slsa-github-generator` runs in a separate GitHub Actions job that cannot be modified by the build job, ensuring isolation.
- Slack notifications should be terse and actionable. Format: `[shinhan-innoboost-engine] main-deploy → staging | sha=abc1234 | smoke=pass | image=ghcr.io/.../engine:main-abc1234 | runtime=4m12s | <runbook-link>`.
- For PR-check parallelism, the `containerise` step depends on `build`; everything else can run in parallel from the install step.
- For the eval-harness workflow's "incremental on PR" mode, identify changed metrics by inspecting the PR diff for files under `engine/metrics/` or `engine/prompts/`; only run gold-set entries that exercise those metrics. This keeps PR-check time bounded.

## Test Plan

- Test 1: Open a sample PR with no changes → all five workflows pass; PR-check runtime measured.
- Test 2: Open a PR introducing a critical-severity vulnerability (e.g., a known-vulnerable dependency version) → `trivy-scan` should fail; PR cannot merge.
- Test 3: Open a PR with a hardcoded secret pattern → `secret-scan` should fail; PR cannot merge.
- Test 4: Open a PR with a non-conventional commit message → `commit-validation` should fail.
- Test 5: Merge a PR to `main` → `main-deploy` runs; image pushed to GHCR; staging deployed; smoke-test passes; Slack notification posted.
- Test 6: Push a `v0.1.0` tag → `release.yml` produces SLSA provenance; cosign verification passes; SLSA verifier passes.
- Test 7: Verify the nightly `security-scan` runs and reports cleanly (or with acknowledged findings only).
- Test 8: Provoke an emergency-merge scenario (a critical CVE that requires immediate patch); verify the documented emergency process is followed and audited.

## Rollback Plan

- A bad pipeline change is rolled back via revert in `main`; the previous main-deploy will recover staging.
- A bad staging deploy is rolled back via `helm rollback` (recorded in the runbook); the previous image is still in GHCR.
- A bad release tag is handled by tagging a fix and re-releasing; old release tags remain (for audit), but are marked superseded.
- An emergency-merge bypass is logged with founder+tech-lead 2-eye record and audited within 24 hours.

## Operational Risks

- **GHCR outage.** Mitigation: image retention in a fallback registry (e.g., GCP Artifact Registry mirror); not blocking demo work, only release pipeline.
- **GitHub OIDC misconfiguration.** Mitigation: test in staging environment first; have a fallback long-lived kubeconfig (sealed) for emergency.
- **Sigstore Rekor outage.** Mitigation: signature still works locally; verification can pull from the local cache; release pipeline retries.
- **Pipeline runtime regression (median > 12 min).** Mitigation: cache profiling; identify the slow step; optimise. Allowed degradations: doubling of test count adds runtime proportionally; resolve by parallelisation.
- **Snyk or Semgrep false-positive flood blocking real PRs.** Mitigation: triage on first run; suppress with documented justifications; never blanket-disable a rule.
- **Provenance verification fails on a release.** Mitigation: investigate immediately; provenance failure is treated as a release-blocker.
